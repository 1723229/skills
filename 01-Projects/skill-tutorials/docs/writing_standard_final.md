# Skill Tutorial Writing Standard - 技能教程写作规范

> 本规范定义技能教程的深度标准、结构要求和写作风格，确保所有教程达到统一的高质量水平。

---

## 一、内容深度标准

### 1.1 信息密度要求

| 维度 | 深度要求 | 说明 |
|------|----------|------|
| **概念解释** | 必须解释"为什么" | 不仅说明功能，还要解释设计意图和适用边界 |
| **操作步骤** | 必须可复现 | 每个步骤都有明确的命令/代码，读者跟着做能成功 |
| **参数说明** | 必须完整 | 所有参数都有类型、必填、默认值、说明四要素 |
| **示例代码** | 必须可运行 | 提供完整的、经过测试的代码示例，不是伪代码 |
| **故障排除** | 必须有场景** | 基于真实错误场景，提供诊断思路和解决方案 |

### 1.2 禁止的浅层内容

❌ **禁止以下浅层写法：**

- "详情请参考官方文档"（没有自己的解释和示例）
- "配置很简单，按照提示操作即可"（没有具体步骤）
- "参数说明见源码"（没有文档化）
- "遇到问题请自行搜索"（没有提供解决方案）
- 只有功能列表，没有使用场景说明

## 二、标准章节结构

每篇教程必须包含以下章节（可根据技能复杂度适当调整）：

### 2.1 概述（Overview）

**必须包含：**
- **功能简介**（150字以上）
  - 这个技能解决了什么问题？
  - 核心能力是什么？
  - 技术实现原理简述
  
- **适用场景**（至少3个具体场景）
  - 每个场景包含：场景描述 + 具体示例
  - 示例："知识沉淀：将项目笔记从 Markdown 迁移到飞书知识库，支持版本管理和团队协作"

- **不适用场景**（至少2个）
  - 明确告知用户什么时候不该用这个技能
  - 示例："不适用：需要实时协作编辑的场景（飞书文档本身支持更好）"

- **依赖要求**
  - Python版本及原因
  - 系统依赖（如需要）
  - 外部服务及获取方式
  - 权限要求

### 2.2 安装配置（Setup）

**必须包含：**
- **环境检查清单**
  - 命令 + 预期输出 + 异常情况处理

- **详细安装步骤**
  - 每个命令的完整形式
  - 常见安装错误的解决方案
  - 验证安装成功的方法

- **配置文件详解**
  - 每个配置项的作用
  - 从哪里获取配置值（带链接或路径）
  - 配置示例（脱敏后的真实配置）
  - 不同环境的配置差异（开发/生产）

- **首次运行验证**
  - 最小可用测试
  - 预期输出
  - 常见问题

### 2.3 核心概念（Concepts）【可选但推荐】

对于复杂技能，必须先解释核心概念：
- 架构图（文字描述或ASCII图）
- 关键术语定义
- 工作流程说明
- 与其他组件的关系

### 2.4 快速开始（Quick Start）

**必须包含：**
- **最小可用示例**（复制粘贴即可运行）
- **分步说明**
  - 每一步的预期结果
  - 截图描述（如果无法截图，用文字描述界面状态）
- **成功验证**
  - 如何判断操作成功
  - 成功后的输出示例

### 2.5 详细用法（Detailed Usage）

**必须包含：**
- **所有功能点**
  - 每个功能独立小节
  - 功能说明 + 使用场景 + 代码示例

- **API/函数完整说明**
  - 函数签名
  - 参数表格（名称、类型、必填、默认值、说明）
  - 返回值表格（名称、类型、说明）
  - 异常说明（异常类型、触发条件、处理建议）

- **配置选项**
  - 所有可配置项
  - 配置对行为的影响
  - 推荐配置组合

### 2.6 示例代码（Examples）

**必须提供3个层次的示例：**

1. **基础示例**（最简单的用法）
   - 单一功能演示
   - 代码行数：10-20行
   - 无外部依赖

2. **进阶示例**（组合用法）
   - 多个功能组合
   - 代码行数：30-50行
   - 展示最佳实践

3. **完整工作流**（生产级用法）
   - 端到端场景
   - 代码行数：50-100行
   - 包含错误处理
   - 包含日志记录
   - 可扩展为实际项目

**每个示例必须包含：**
- 前置条件说明
- 完整可运行代码
- 代码逐行注释
- 预期输出
- 运行结果验证方法

### 2.7 最佳实践（Best Practices）

**必须包含：**
- 性能优化建议
- 安全注意事项
- 常见陷阱及避免方法
- 推荐的工作流
- 与其他技能的组合使用

### 2.8 故障排除（Troubleshooting）

**必须包含：**
- **错误场景分类**
  - 安装问题
  - 配置问题
  - 运行时问题
  - 性能问题

- **每个错误必须提供：**
  - 错误现象（完整错误信息）
  - 根因分析（为什么会发生）
  - 诊断步骤（如何定位）
  - 解决方案（一步一步修复）
  - 预防措施（如何避免再次发生）

- **调试技巧**
  - 日志开启方法
  - 常用调试命令
  - 日志分析方法

### 2.9 参考链接（References）

**必须包含：**
- 源码路径（相对路径）
- 相关的其他技能链接
- 外部官方文档链接
- 推荐阅读顺序

## 三、写作风格指南

### 3.1 语言风格

- **专业但易懂**：技术术语必须解释，或提供术语表
- **第二人称**：使用"你"直接对读者说话
- **主动语态**："运行这个命令" 而不是 "这个命令应该被运行"
- **具体明确**：避免"可能"、"也许"，用确定的表述

### 3.2 代码规范

- 所有代码块必须指定语言
- 代码必须有注释（特别是关键步骤）
- 示例代码必须完整，不能省略关键部分
- 命令行示例必须包含提示符（$ 或 #）
- 输出示例必须清晰标注

### 3.3 视觉呈现

- 使用表格呈现对比信息（3项以上）
- 使用列表呈现步骤（7步以内）
- 使用代码块呈现命令和代码
- 使用引用块呈现注意事项和提示

## 四、质量检查清单

发布前必须检查：

- [ ] 所有代码示例经过实际测试，可运行
- [ ] 所有链接可访问
- [ ] 所有命令有预期输出说明
- [ ] 故障排除覆盖至少5个常见错误
- [ ] 有完整的从零开始到成功的路径
- [ ] 术语有解释或链接到解释
- [ ] 不同水平读者都能找到适合的起点

## 五、示例对比

### ❌ 浅层写法（禁止）

```markdown
## 安装

安装依赖：

```bash
pip install requests
```

然后配置环境变量即可使用。
```

### ✅ 深度写法（标准）

```markdown
## 安装

### 1. 环境检查

确保 Python 版本 >= 3.8：

```bash
$ python3 --version
Python 3.9.7  # 符合要求（>= 3.8）
```

如果版本过低，请升级 Python：
- macOS: `brew install python@3.9`
- Ubuntu: `sudo apt update && sudo apt install python3.9`
- Windows: 从官网下载安装包

### 2. 安装依赖

```bash
$ pip3 install requests python-dotenv
```

**为什么需要这些依赖？**
- `requests`: 用于调用飞书 API
- `python-dotenv`: 用于从 .env 文件加载配置

**验证安装成功：**

```bash
$ python3 -c "import requests; print(requests.__version__)"
2.28.1
```

如果提示 `ModuleNotFoundError`，说明 pip 安装路径不在 PYTHONPATH 中，请检查：

```bash
$ which python3
$ which pip3
```

确保两者属于同一个 Python 安装。

### 3. 复制技能文件

```bash
$ cp -r skills/feishu-doc-orchestrator ~/.openclaw/workspace/skills/
```

**预期结果**：
```bash
$ ls ~/.openclaw/workspace/skills/feishu-doc-orchestrator/
SKILL.md  scripts/  feishu-md-parser/  ...
```
```

## 六、执行计划

按本规范重写教程的流程：

1. **技能分析**（每篇30分钟）
   - 阅读源码和SKILL.md
   - 理解核心功能和工作流程
   - 识别关键概念和潜在陷阱

2. **示例开发**（每篇60分钟）
   - 编写并测试基础示例
   - 编写并测试进阶示例
   - 编写并测试完整工作流
   - 记录常见错误和解决方案

3. **文档编写**（每篇90分钟）
   - 按规范撰写各章节
   - 插入代码示例和表格
   - 检查所有链接和引用

4. **质量审核**（每篇30分钟）
   - 对照检查清单
   - 测试所有代码
   - 修正发现的问题

**预估时间**：每篇深度教程约 **3.5小时**，49篇总计约 **170小时**。

建议：先完成**核心高频技能**（约10篇）的深度教程，其余保持基础框架，后续迭代完善。
